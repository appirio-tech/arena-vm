The admin listener server has the following queues:

·             One for incoming messages

·             One for calls to the EJB

 

The queues were made separate to facilitate their handling by different threads.

 

An active admin listener server has the following threads running:

·             One for the contest server reader (if connected).  Managed by ContestServerConnection using a SocketReaderThread.

·             One reader thread for each active admin monitor client connection.  Managed by ClientConnection using its own thread code.

·             One reader thread waiting for the next admin monitor client connection, also managed by ClientConnection using its own thread code.

·             One dispatch thread which pulls items from the incoming message queue and sends them on to their destination.  This destination can be either an admin monitor client, all monitor clients (a broadcast), the contest server, or the EJB call queue.  Managed by AdminProcessor using a QueueReaderThread.

·             One dispatch thread which pulls items from the EJB call queue, invokes the appropriate method in the EJB, and places a response back on the incoming message queue.  Because this thread is separate from the other dispatch thread, a slow EJB call will not interfere with traffic between the monitor clients and the contest server.  Managed by BackEndDispatcher using a QueueReaderThread.

·             One logging server socket thread that listens for incoming logging streams from other server components (including the admin listener itself.)

·             One logging worker thread for each incoming stream.  Each incoming LoggingEvent is retransmitted to each subscribed client connection.

·             One logging thread for the admin listener’s own TCSocketAppender.  This thread exists in all server components that use remote logging, and allows for logging to take place asynchronously.

 

Connections are identified by a connection ID, which is always 5 for the contest server connection (note that at this time, only one simultaneous contest server connection is supported) and are sequentially numbered starting at 10 for admin monitor client connections.  The ClientConnectionSet keeps track of which monitor client connections are active at any point in time.

 

Running the server

 

There is a script adminListener.sh in the scripts directory that works similarly to listener.sh.

 

adminListener.sh (start|stop|kill) [port] [contest listener IP:port] [logging port]

            start - start listener in the background

            stop  - stop listener

            kill  - kill listener with 'kill -3'

 

When starting up the admin listener or an admin monitor client, it will take a bit of time owing to the necessity of initializing a cryptographically strong source of random numbers, which is used by the SSL connection.  This process takes around 35 seconds on my home PC.  Thus, have some patience.

 

Note that when running everything all at once on a machine, there are several different ports that will be in use:

-         The port on which the contest listener listens for client connections

-         The port on which the contest listener listens for the admin server connection.  This should only be accessible from within the TC network.

-         The port on which the admin listener listens for admin monitor client connections.  The port on which the logging server listens for incoming streams.  This should only be accessible from within the TC network.

 

The admin listener does not require the contest listener to be up to run.  It does, however, require the database server to be up on startup.  If the contest listener server is brought up after the admin listener has been started, a command will need to be issued from an admin monitor client telling the admin listener to connect to the contest server.

 

A sequence to start everything running could look something like this:

First, if the database server is not running, start weblogic on the 65.112.118.219 box:

cd weblogic_commands

start

 

Then start the contest listener, the admin listener, on server:

cd

cd app/scripts

listener.sh start 5000 d 5100

adminListener.sh start 5200 172.16.20.30:5100 5013

 

 

Then start the monitor app on your local machine:

cd app\scripts

monitor_gui.bat 65.112.118.207:5200

            or

cd app/scripts

monitor_gui.sh 65.112.118.207:5200.
